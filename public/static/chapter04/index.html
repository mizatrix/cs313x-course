Hardware Basics: The I/O Bottleneck

Information Retrieval is I/O Bound — speed is limited by disk, not CPU.

▶ Professor Notes

🎤 What to Say

"Before we build any index, we need to understand WHY all these algorithms exist. It all comes down to one problem: disk is SLOW. Everything we learn today is about working around that bottleneck."

🎯 Analogy

Think of seek time like physically driving to a library in another city (slow!), and transfer time like reading a book once you're there (fast). Our goal is to minimize the number of trips.

🙋 Expected Student Questions

"Why not just use SSDs for everything?"

Great question! SSDs are 100x faster than HDDs, but for petabyte-scale indexes (Google), even SSD seek time adds up. Plus, RAM is still 100x faster than SSD. The real goal is to fit the index in RAM.

"Is this still relevant with modern hardware?"

Absolutely. The speed gaps between memory levels haven't changed much. L1 cache is still ~10,000x faster than SSD. The principles are timeless.

🔗 Why This Matters in IR

Every IR system — Google, Elasticsearch, Solr — is fundamentally limited by I/O. The entire chapter exists because of this bottleneck. BSBI, SPIMI, compression, distributed indexing — they're all strategies to work around slow disk. Students who understand this will understand WHY all the other algorithms were invented, not just HOW they work.

💡 Transition

"Now that we know disk is the bottleneck, let's see how BSBI solves the problem of building an index when data doesn't fit in RAM..."

02

BSBI: Block Sort-Based Indexing

External sorting when data doesn't fit in RAM.

▶ Professor Notes

🎤 What to Say

"Imagine you have 100GB of text but only 16GB of RAM. You can't sort everything at once. BSBI's idea is simple: break it into chunks that fit in memory, sort each chunk, save to disk, then merge them all at the end."

🎯 Analogy

It's like sorting 1000 playing cards when your table only fits 100. You sort 100 at a time, put each sorted pile aside, then merge all piles by taking the smallest card from each pile's top.

🙋 Expected Student Questions

"What is an inverted index exactly?"

It's like the index at the back of a textbook. Instead of 'page → words', it's 'word → list of documents containing it'. That's the postings list.

"Why do we sort by TermID?"

So that when we merge, all occurrences of the same term end up together. This is what makes the merge step work — it's like merge sort.

"What's a file handle?"

Each open file uses a 'file handle' — an OS resource. If you have 10,000 segments, you need 10,000 file handles open simultaneously during merge. Operating systems have limits on this.

🔗 Why This Matters in IR

BSBI introduces the fundamental concept of external sorting — how to process data that doesn't fit in memory. This is the foundation of all large-scale indexing. Without understanding BSBI, students can't appreciate why SPIMI is an improvement, or why distributed systems like MapReduce are needed. It also introduces the inverted index structure that is the backbone of ALL search engines.

💡 Transition

"BSBI works, but it wastes memory by storing every (term, docID) pair separately. What if the term 'the' appears 1 million times? That's 1 million copies of the same TermID. SPIMI fixes this..."

03

SPIMI: Single-Pass In-Memory Indexing

Builds postings lists directly — no pairs, no sorting!

▶ Professor Notes

🎤 What to Say

"SPIMI is the smarter cousin of BSBI. Instead of storing 'the, doc1', 'the, doc2', 'the, doc3' as separate pairs, SPIMI maintains a dictionary in RAM and just appends doc IDs directly: 'the → [1, 2, 3]'. No sorting needed!"

🎯 Key Point to Emphasize

The key insight is that SPIMI uses a hash map (Python dictionary), so lookup is O(1). BSBI had to sort because it couldn't look up terms quickly in an unsorted list of pairs.

🙋 Expected Student Questions

"Why is SPIMI faster?"

Two reasons: (1) No O(N log N) sorting step — just O(N) appends, and (2) uses less memory because each term is stored only once in the hash map, not repeated in every pair.

"Which one do real systems use?"

SPIMI! Lucene (the engine behind Elasticsearch) uses a SPIMI-like approach. BSBI is mainly taught for historical context and to understand why SPIMI is better.

🔗 Why This Matters in IR

SPIMI is not just history — it's how Lucene actually works today. When Elasticsearch indexes a document, it uses SPIMI-like in-memory buffering, then flushes immutable segments to disk. Understanding SPIMI means understanding the core of every modern search engine. It also introduces the concept of dynamic dictionaries, which connects to how real-time indexing handles new terms.

💡 Transition

"Both BSBI and SPIMI work on a single machine. But what if you need to index the entire web — petabytes of data? You need distributed processing. Enter MapReduce..."

04

Distributed Indexing: MapReduce

Scaling from one machine to thousands — the Google way.

▶ Professor Notes

🎤 What to Say

"Google published the MapReduce paper in 2004 and it revolutionized data processing. The idea is beautifully simple: split the work into two phases — Map (parse and extract) and Reduce (aggregate and build). Anyone can understand it."

🎯 Analogy

Imagine counting word frequencies in 1000 books. Map phase: give each student 10 books and ask them to list every word. Shuffle: collect all lists and group by word. Reduce: one student per letter counts the totals for their words.

🐳 Docker Demo Instructions

When you reach this section, open a terminal and run:

  • Make sure Docker Desktop is running first!
  • cd docker-mapreduce && docker compose up --build
  • Watch the colored logs — mappers tokenize → coordinator shuffles → reducers build index
  • When done: docker compose down

🙋 Expected Student Questions

"Why is Shuffle the most expensive phase?"

Because it requires moving ALL the mapper outputs across the network to the right reducer. If you have 1000 mappers, each producing 1GB of output, that's 1TB of data being transferred over the network.

"Is MapReduce still used today?"

The original Hadoop MapReduce is less common now. Modern systems use Spark or Flink which keep data in memory. But the MAP → SHUFFLE → REDUCE pattern is fundamental and appears everywhere.

"What's the difference between Term and Document partitioning?"

Term partitioning: split the dictionary (A-M on server 1, N-Z on server 2). Document partitioning: split the documents (docs 1-1M on server 1, 1M-2M on server 2). Document partitioning is the industry standard because it balances load better.

🔗 Why This Matters in IR

MapReduce is the bridge from single-machine IR to web-scale IR. Without distributed indexing, Google couldn't index 100 billion web pages. This concept also connects to how Elasticsearch distributes shards across nodes — same principle, modern implementation. Understanding partitioning strategies (term vs document) is critical because it determines query latency, load balance, and fault tolerance in production search systems.

💡 Transition

"Now we have our index, but it might be 10x larger than necessary. Compression is not just about saving disk — it's about making search FASTER. Let me show you why..."

🖥️ Single Machine Limits

  • ❌ Storage Cap: Can't fit PB of data
  • ❌ Slow Indexing: Weeks to index the web
  • ❌ Single Point of Failure

☁️ Distributed Cluster

  • ✅ Scalability: Add cheap nodes
  • ✅ Parallelism: Index in hours
  • ✅ Fault Tolerance: Survives failures

Index Compression

Smaller index = faster search. The big win: fitting the index in RAM.

▶ Professor Notes

🎤 What to Say

"Most students think compression is about saving disk. Wrong! The REAL benefit is speed. A compressed index that fits in RAM avoids disk seeks entirely. Remember slide 1 — disk seeks are 10,000x slower than RAM access."

🎯 VB Encoding — How to Explain

Think of each byte as a letter in an envelope. 7 bits carry your actual message, and 1 bit (the stamp) tells you if there are more envelopes coming (0) or if this is the last one (1). That's Variable Byte encoding.

🎯 Gap Encoding — How to Explain

Instead of storing 'I live at house 33, 45, 52, 90, 120', store 'I start at 33, then go +12, +7, +38, +30'. Gaps are much smaller numbers and need fewer bits.

🙋 Expected Student Questions

"Why not always use Gamma (better compression)?"

Because modern CPUs are so fast that the bottleneck shifts to DECODING speed. VB is byte-aligned so the CPU can process it naturally. Gamma requires bit-level manipulation which is much slower. In practice, decoding speed wins.

"What do real systems actually use?"

Lucene/Elasticsearch use a technique called PForDelta — it processes 128 integers at once using SIMD CPU instructions. It's 5x faster than VB! But for exams, you need to know VB and Gamma.

🔗 Why This Matters in IR

Compression is arguably the highest-impact optimization in IR. A 4x compression ratio means the index fits in RAM instead of disk — that's a 10,000x speedup for random access! Gap encoding + VB encoding is the standard approach used in Lucene, Elasticsearch, and Solr. Without compression, modern search engines would need 4x more servers — billions of dollars in infrastructure costs at Google's scale.

⚠️ Exam Tips

VB encoding of specific numbers is a common exam question. Make sure students can encode 824 and decode [0x06, 0xB8] by hand. Use the interactive demo to let them practice!

💾 Disk Space

Reduces physical footprint, lowers storage costs.

⚡ I/O Speed

Reading compressed data from disk is faster even with decompression.

🧠 Caching (RAM)

The Big Win: Compressed index fits in memory → avoids slow disk seeks!

06

Query Processing

Phrase queries, wildcards, skip pointers and spelling correction.

▶ Professor Notes

🎤 What to Say

"Now we have a compressed, distributed index. The last piece of the puzzle is: how do we actually ANSWER queries? We need to intersect postings lists efficiently, handle phrase queries where word ORDER matters, and deal with user typos."

🎯 Skip Pointers — How to Explain

Imagine looking for common friends between two sorted friend lists. Without skips, you compare name by name. With skip pointers, when you see 'Alice' in list A and 'Zara' in list B, you can jump ahead in list A past all names before Z. Saves tons of comparisons.

🎯 Permuterm — How to Explain

The trick is: to search for 'h*o', we rotate the pattern so the wildcard is at the end: 'o$h*'. Now it's just a prefix search! We pre-compute all rotations of every term so any wildcard can become a prefix query.

🙋 Expected Student Questions

"Why √L for skip pointer spacing?"

It's a mathematical optimization. Too few skips = no benefit. Too many skips = you check a skip at every element. √L minimizes the total expected comparison cost. It's similar to the optimal number of floors to test with 2 eggs.

"Why not just use positional indexes everywhere?"

Because storing positions makes the index 2-4x larger. For a term like 'the' that appears billions of times, storing every position is extremely expensive. Many systems use a hybrid: positional for important terms, bi-word for common phrases.

🔗 Why This Matters in IR

Query processing connects ALL previous concepts: the index we built (BSBI/SPIMI), compressed (VB/Gamma), and distributed (MapReduce) now needs to answer user queries efficiently. Skip pointers directly reduce query latency. Phrase queries enable exact-match search ("to be or not to be"). Wildcards enable exploratory search. Spelling correction improves recall. Every concept here directly impacts the user experience of a search engine.

07

Real-World Systems

Lucene, Elasticsearch, and Twitter Earlybird.

▶ Professor Notes

🎤 What to Say

"Everything we learned today — SPIMI, compression, segments, merging — this is EXACTLY how Lucene works. Elasticsearch is just Lucene + distribution + REST API. If you understand this chapter, you understand the core of every major search engine."

🎯 Key Points

Lucene = engine (core indexing library). Elasticsearch = car (makes it distributed, adds REST API, clustering). Think of it like SQLite (library) vs PostgreSQL (server).

  • Segments are immutable — once written, never changed. This eliminates lock contention and makes search fast.
  • Deletes use tombstones — a .liv bitset marks deleted docs. Physical deletion only happens during a merge.
  • Twitter's Earlybird — keeps the active segment in RAM for real-time search. Tweets are searchable within seconds because they never wait for disk.

🙋 Expected Student Questions

"What's a shard in Elasticsearch?"

A shard is a single Lucene instance. An index (e.g., 'products') is split into N shards distributed across nodes. Each shard is a complete, self-contained inverted index.

"How does Elasticsearch handle node failures?"

Each shard has replicas on other nodes. If Node 1 dies, the replica on Node 3 is promoted to primary. Zero downtime. This is why the diagram shows P1/R1 on different nodes.

🔗 Why This Matters in IR

This section ties EVERYTHING together. Lucene uses SPIMI + immutable segments + VB compression. Elasticsearch adds MapReduce-style distribution + document partitioning. Twitter's Earlybird shows real-time indexing with auxiliary indexes. Students should leave understanding that every concept we taught is used in production systems processing billions of queries daily. This isn't theory — this is how the internet works.
